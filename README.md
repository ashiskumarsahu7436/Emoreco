# EMORECO

A full-stack voice emotion recognition platform built with React, Node.js, and PostgreSQL — powered by Deepgram, Behavioral Signals, and Groq.

> Built to explore where speech AI meets behavioral psychology: real-time transcription, acoustic emotion extraction from vocal patterns, and LLM-generated psychological insight — all from a single audio recording.

---

## What It Does

EMORECO lets users record or upload audio and receive a deep, AI-generated psychological breakdown of the emotions present in the speaker's voice — going beyond simple sentiment labels to explain what the person felt, why they likely felt it, and what they were truly trying to express.

**Core user flows:**
- **Record live audio** from the microphone with a real-time animated waveform and volume/clarity meters, or **upload an existing file** (WAV, MP3, M4A, WebM, OGG)
- Receive a full **voice profile** from Behavioral Signals: dominant emotion, sentiment, vocal strength, speaking rate, hesitation detection, speaker gender, estimated age range, and detected language
- Get a **2–3 sentence primary emotion summary** and a **full psychological paragraph** generated by Groq (Llama 3.3 70B) — grounded in both the transcription and the acoustic voice data
- **Chat with the AI** about any analysis — ask follow-up questions about the speaker's emotions, intent, or patterns; the full conversation is saved per analysis
- **Export a PDF report** of the analysis streamed directly from the server as a file download
- Group analyses into **Spaces** — named containers tied to a specific person; the AI draws on past Space analyses for richer longitudinal insight
- Browse the full **Analysis History** with filtering by Source (recordings vs. uploads), Sentiment (positive/negative), Space, and free-text search
- **Delete** individual analyses or bulk-clear all analyses in a Space

## Database Schema

Defined and auto-created in `server/config/database.js` via `initDb()` (lines 10–47). Tables are created with `CREATE TABLE IF NOT EXISTS` on every server start.

| Table | Lines | Description |
|---|---|---|
| `users` | 12–19 | `id SERIAL PK`, `name`, `email UNIQUE`, `password` (bcrypt), `created_at` |
| `spaces` | 21–28 | `id SERIAL PK`, `user_id FK→users CASCADE`, `name`, `person_name`, `created_at` |
| `analyses` | 30–43 | `id SERIAL PK`, `user_id FK→users CASCADE`, `space_id FK→spaces SET NULL`, `audio_path` (prefixed `mic:` or `upload:`), `transcription`, `language`, `hume_emotions` (JSON string of full voice profile), `primary_emotion`, `detailed_analysis`, `chat_history` (JSON array), `created_at` |

Three indexes are created alongside: `idx_analyses_user_id`, `idx_analyses_space_id`, `idx_spaces_user_id` (lines 44–46).

## Key Technical Decisions

### Parallel Deepgram + Behavioral Signals (`analysisController.js` lines 41–44)
Transcription (Deepgram) and acoustic analysis (Behavioral Signals) are fired together using `Promise.allSettled`. Since Behavioral Signals polling takes 30–160 seconds, running them in parallel cuts total wait time roughly in half. `allSettled` instead of `all` means a Behavioral Signals failure still lets the transcription succeed — the AI analysis degrades gracefully (no voice profile, but the Groq prompts still run) instead of returning a 500.

### Behavioral Signals Flat-Array Parsing (`parseBehavioralResults`, lines 257–356)
The v5 API returns a **flat array** where each entry has a `task` field (e.g. `"emotion"`, `"asr"`, `"gender"`), a `level` field, and a `finalLabel`. The parser filters to `level === 'utterance'` only — skipping file-level aggregates — tallies label frequency per task across all utterances, and picks the most common label as dominant. Key quirks: `asr.finalLabel` is a plain transcription string (not an object); `age.finalLabel` is a range string like `"31 - 45"` (not a number).

### Original Filename Required for Behavioral Signals (`analysisController.js` line 43)
Multer saves temp files without a file extension (e.g. `uploads/a3f2bc91d...`). Behavioral Signals v5 infers audio format from the submitted filename — without an extension the API returns a 500 error. `req.file.originalname` is passed as the second argument to `analyzeBehavioralSignals` so the correct MIME type is set on the `FormData` blob and the filename carries the right extension.

### Space History as LLM Context (`analyzeAudio` lines 63–87)
When an analysis is submitted to a Space, the controller fetches the 5 most recent prior analyses in that Space before calling Groq. The primary emotion prompt receives all 5 as a numbered list of past summaries. The detailed analysis prompt receives the 2 most recent (each truncated to 200 chars). This gives the model enough longitudinal context to comment on emotional patterns without exhausting the token budget.

### `?` → `$N` SQL Adapter (`database.js` lines 73–76)
The codebase originally used SQLite (which uses `?` for parameters). All SQL strings across the project still use `?`. The `toPositional()` function converts them to `$1, $2, …` using a closure counter before every `pg` query — so the migration to PostgreSQL required zero changes to any query string.

### JWT Over Sessions
Auth uses stateless JWT tokens (30-day expiry, signed with `JWT_SECRET`) stored by the client in `localStorage`. This removes the need for a session table or Redis store, which simplifies Render deployment: no extra service, no session migration, no sticky sessions.

## Authentication

Email + password flow (`server/routes/auth.js`):

1. User registers → `POST /api/auth/signup` (line 8) — validates fields, checks for duplicate email, bcrypt-hashes the password, inserts user row, returns a signed JWT + `{ id, name, email }`
2. User logs in → `POST /api/auth/login` (line 43) — `bcrypt.compare` validates password, returns a signed JWT + user object
3. All protected routes pass through `authenticateToken` middleware (`server/middleware/auth.js`) — reads the `Authorization: Bearer <token>` header, calls `jwt.verify`, attaches `req.user = { userId, email }`, returns 401 if missing or 403 if invalid/expired

```

Spaces                                   (all routes require Bearer token)

Health
  GET    /health                         { status: 'ok', message: 'EMORECO API is running' }
```

## Running Locally

**Prerequisites:** Node.js 18+, a Supabase project (or any PostgreSQL instance), accounts on Deepgram, Groq, and Behavioral Signals.

```bash
# 1. Clone the repo
git clone https://github.com/your-username/emoreco.git
cd emoreco

# 2. Install dependencies
npm install
cd client && npm install && cd ..
cd server && npm install && cd ..

# 3. Create server/.env with your credentials
DATABASE_URL=postgresql://postgres:[password]@[host]:6543/postgres
JWT_SECRET=a-random-string-at-least-32-characters-long
DEEPGRAM_API_KEY=your-deepgram-key
GROQ_API_KEY=your-groq-key
BEHAVIORAL_SIGNALS_CID=your-client-id
BEHAVIORAL_SIGNALS_TOKEN=your-auth-token

# 4. Start both servers (Express on :3000, Vite on :5000)
npm run dev
```

Open `http://localhost:5000`. Vite proxies all `/api/*` requests to Express on port 3000 — no CORS configuration needed in development.

## Deploying to Render

**Build command:** `npm install && npm run build`
**Start command:** `npm start`

In production, Express serves the compiled React app from `client/dist` on the same origin — no separate static host or CORS setup required.

Required environment variables on Render:

| Variable | Value |
|---|---|
| `DATABASE_URL` | Supabase PostgreSQL connection string |
| `JWT_SECRET` | Random string, minimum 32 characters |
| `DEEPGRAM_API_KEY` | From console.deepgram.com |
| `GROQ_API_KEY` | From console.groq.com |
| `BEHAVIORAL_SIGNALS_CID` | Your Behavioral Signals client ID |
| `BEHAVIORAL_SIGNALS_TOKEN` | Your Behavioral Signals auth token |
| `NODE_ENV` | `production` |
| `PORT` | `3000` |
